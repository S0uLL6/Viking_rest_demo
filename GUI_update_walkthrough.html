Обновление GUI после REST — по шагам

Три сценария: POST (создание), DELETE (удаление), PUT (обновление). Каждый — пять шагов от прихода HTTP-запроса до перерисованной строки в таблице. Подсвеченные жёлтым строки — ключевые для понимания.

Структура одинаковая для всех трёх: Controller → Listener → invokeLater → Frame → TableModel → JTable. Меняется только конкретный метод на каждом уровне.

POST Сценарий 1 — создание викинга

1 REST-запрос приходит в контроллер VikingController.java

Spring видит POST на /api/vikings, метод createViking. Джексон превращает JSON в объект Viking (без id — клиент его не присылает). Контроллер зовёт сервис. 

@PostMapping
@Operation(summary = "Добавить конкретного викинга", operationId = "createViking")
public ResponseEntity<Viking> createViking(@RequestBody Viking viking) {
    System.out.println("POST /api/vikings called with body: " + viking);
    Viking created = vikingService.create(viking);   // сохранили в БД, получили объект с id
    vikingListener.onCreated(created);                // → шаг 2
    URI location = URI.create("/api/vikings/" + created.id());
    return ResponseEntity.created(location).body(created);
}
↓
2 Listener оборачивает в invokeLater VikingListener.java

Мы сейчас в потоке Tomcat. Swing требует EDT. SwingUtilities.invokeLater ставит лямбду в очередь EDT — Swing сам её выполнит когда сможет. 

/** REST-операция POST успешна — добавляем новую строку в таблицу. */
public void onCreated(Viking viking) {
    if (gui != null) {
        SwingUtilities.invokeLater(() -> gui.addNewViking(viking));   // → шаг 3 (но в EDT)
    }
}
↓
3 Frame передаёт правку модели VikingDesktopFrame.java

Frame не лезет в данные сам, он просто пробрасывает викинга в tableModel.

public void addNewViking(Viking viking) {
    tableModel.addViking(viking);   // → шаг 4
}
↓
4 TableModel добавляет в список и оповещает JTable VikingTableModel.java

Кладём викинга в конец списка и говорим JTable: «вставлена строка с индексом row». JTable перерисует ТОЛЬКО её, не трогая остальные. 

public void addViking(Viking viking) {
    int row = data.size();
    data.add(viking);                       // добавили в список
    fireTableRowsInserted(row, row);        // → шаг 5 (JTable получает уведомление)
}
↓
5 JTable рисует новую строку встроенный механизм Swing

Swing-таблица слушает события модели. fireTableRowsInserted вызывает tableChanged у JTable, и она запрашивает значения только для новой строки через getValueAt(row, col). Никаких других строк не трогает.

✓ Викинг создан в БД, строка добавлена в окно, существующее выделение и прокрутка сохранены.

DELETE Сценарий 2 — удаление викинга

1 Контроллер удаляет из БД и зовёт листенер VikingController.java

id берётся из URL через @PathVariable. Сервис удаляет викинга (equipment удалится каскадом через FK). Дальше — обновить GUI по этому id. 

@DeleteMapping("/{id}")
@Operation(summary = "Удалить викинга по id", operationId = "deleteViking")
public ResponseEntity<Void> deleteViking(@PathVariable int id) {
    System.out.println("DELETE /api/vikings/" + id + " called");
    vikingService.deleteById(id);     // удалили из БД
    vikingListener.onDeleted(id);     // → шаг 2
    return ResponseEntity.noContent().build();
}
↓
2 Listener в EDT VikingListener.java 
/** REST-операция DELETE успешна — удаляем конкретную строку по id. */
public void onDeleted(int id) {
    if (gui != null) {
        SwingUtilities.invokeLater(() -> gui.removeViking(id));
    }
}
↓
3 Frame пробрасывает в tableModel VikingDesktopFrame.java 
/** Удалить строку с викингом по id (точечное обновление таблицы). */
public void removeViking(int id) {
    tableModel.removeByVikingId(id);
}
↓
4 TableModel ищет по id и удаляет одну строку VikingTableModel.java

Бежим по списку, находим викинга с нужным id (у каждого Viking есть поле id, поле прячем от JTable но используем внутри). Удаляем элемент и оповещаем таблицу. 

/** Удаляет строку с викингом, у которого данный id. */
public void removeByVikingId(int id) {
    for (int i = 0; i < data.size(); i++) {
        Viking v = data.get(i);
        if (v.id() != null && v.id() == id) {
            data.remove(i);                  // убрали из списка
            fireTableRowsDeleted(i, i);      // JTable: «удалена строка i»
            return;
        }
    }
}
↓
5 JTable убирает строку встроенный механизм Swing

JTable получает событие удаления и убирает конкретную строку. Все строки ниже сдвигаются вверх автоматически — мы об этом не заботимся.

✓ Викинг удалён из БД, строка пропала из окна, остальные строки на местах.

PUT Сценарий 3 — обновление викинга

1 Контроллер обновляет в БД и зовёт листенер VikingController.java

Два источника: id из URL, тело викинга из JSON. Сервис делает SQL UPDATE и возвращает уже обновлённого викинга. Если такого id в БД нет — кидаем 404 (тут листенер уже не дёргается). 

@PutMapping("/{id}")
@Operation(summary = "Полностью перезаписать параметры викинга")
public Viking updateViking(@PathVariable int id, @RequestBody Viking viking) {
    System.out.println("PUT /api/vikings/" + id + " called with body: " + viking);
    try {
        Viking updated = vikingService.update(id, viking);   // обновили в БД
        vikingListener.onUpdated(updated);                    // → шаг 2
        return updated;
    } catch (NoSuchElementException e) {
        throw new ResponseStatusException(HttpStatus.NOT_FOUND, e.getMessage());
    }
}
↓
2 Listener в EDT VikingListener.java 
/** REST-операция PUT успешна — заменяем строку с этим id новыми данными. */
public void onUpdated(Viking viking) {
    if (gui != null) {
        SwingUtilities.invokeLater(() -> gui.replaceViking(viking));
    }
}
↓
3 Frame пробрасывает в tableModel VikingDesktopFrame.java 
/** Заменить викинга в таблице (находим по id, подставляем новые значения). */
public void replaceViking(Viking viking) {
    tableModel.replaceViking(viking);
}
↓
4 TableModel ищет по id и заменяет VikingTableModel.java

Находим викинга с тем же id что у пришедшего, делаем data.set(i, viking) — кладём новый объект на то же место. fireTableRowsUpdated заставит JTable перерисовать только эту строку. 

/** Заменяет викинга в таблице (находит по id и подставляет новый объект). */
public void replaceViking(Viking viking) {
    if (viking.id() == null) return;
    for (int i = 0; i < data.size(); i++) {
        Viking v = data.get(i);
        if (v.id() != null && v.id().equals(viking.id())) {
            data.set(i, viking);              // заменили объект в списке
            fireTableRowsUpdated(i, i);       // JTable: «обновлена строка i»
            return;
        }
    }
}
↓
5 JTable перерисовывает одну строку встроенный механизм Swing

Самое аккуратное обновление: индекс строки не меняется, прокрутка и выделение сохраняются. Если эта строка была выделена — выделение останется на ней.

✓ Викинг обновлён в БД, строка показывает новые значения, состояние UI не сброшено.
Если препод покажет на код и спросит «объясни этот метод» — у тебя по каждому файлу есть 1–3 ключевых метода, их и нужно знать наизусть:
  • Controller: три эндпоинта — POST, DELETE, PUT — каждый делает свою работу с БД через service, потом дёргает listener.
  • Listener: три метода onCreated/onDeleted/onUpdated — каждый оборачивает действие в SwingUtilities.invokeLater.
  • Frame: три обёртки addNewViking/removeViking/replaceViking — просто проброс в tableModel.
  • TableModel: ключевая логика — поиск по id и оповещение JTable через fireTableRowsXxx.
