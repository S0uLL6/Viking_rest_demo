Аннотации в Spring — шпаргалка

Все аннотации, которые встречаются в практикуме Spring Web, по группам. Каждая карточка: что делает, где у тебя в коде, мини-пример. Сверху более частые и важные, ниже — Swagger-аннотации (они только для документации).

1. Аннотации Spring Web (контроллер и HTTP)

@RestController

Стоит на классе VikingController (строка 25).

Говорит Spring: «этот класс — обработчик HTTP-запросов, а всё, что возвращают его методы, нужно автоматически превращать в JSON и отправлять в теле ответа».

Под капотом это @Controller + @ResponseBody на всех методах. Без него нужно было бы вешать @ResponseBody на каждый метод вручную. 

@RestController
public class VikingController {
    @GetMapping public List<Viking> getAll() { ... }   // вернётся как JSON
}
Когда задают: ответ — «помечает класс как REST-контроллер, ответы автоматически сериализуются в JSON».

@RequestMapping("/api/vikings")

На классе (строка 26).

Задаёт общий префикс пути для всех методов внутри класса. К нему добавляется то, что указано в @GetMapping, @PostMapping и т.д. 

@RequestMapping("/api/vikings")
class VikingController {
    @GetMapping             // → GET  /api/vikings
    @GetMapping("/test")    // → GET  /api/vikings/test
    @DeleteMapping("/{id}") // → DELETE /api/vikings/{id}
}
В практикуме: все эндпоинты пошли через /api/vikings — это и есть «префикс».

@GetMapping / @PostMapping / @PutMapping / @DeleteMapping

На методах контроллера (строки 38, 49, 60, 73, 86, 98).

Сокращённые формы. Каждая привязывает метод к конкретному HTTP-глаголу: 

@GetMapping     ≡ @RequestMapping(method = RequestMethod.GET)
@PostMapping    ≡ @RequestMapping(method = RequestMethod.POST)
@PutMapping     ≡ @RequestMapping(method = RequestMethod.PUT)
@DeleteMapping  ≡ @RequestMapping(method = RequestMethod.DELETE)

Без аргумента — путь совпадает с префиксом класса. С аргументом — добавляется к префиксу.

Внимание: если два метода висят на одном URL+глаголе, Spring при старте упадёт с ошибкой «ambiguous mapping».

@PathVariable

В параметрах: deleteViking(@PathVariable int id), updateViking(@PathVariable int id, ...).

Извлекает значение из пути URL и подставляет в параметр метода. Имена должны совпадать: {id} в маппинге ↔ int id в параметре. Если имена разные — указываем явно: @PathVariable("id") int someId. 

@DeleteMapping("/{id}")
public ResponseEntity<Void> deleteViking(@PathVariable int id) {
    // если URL = DELETE /api/vikings/5 → id = 5
}
Что если прислать /abc? Spring не сможет превратить «abc» в int → отдаст HTTP 400 Bad Request ещё до запуска метода.

@RequestBody

В параметрах: createViking(@RequestBody Viking viking), updateViking(..., @RequestBody Viking viking).

Берёт тело HTTP-запроса (текст JSON), отдаёт его в Jackson, тот десериализует в указанный Java-тип. Поля JSON сопоставляются с полями record-а/класса по именам. 

// HTTP-запрос: POST /api/vikings
// Body: {"name":"Ragnar","age":40, ...}

@PostMapping
public ResponseEntity<Viking> createViking(@RequestBody Viking viking) {
    // viking.name() == "Ragnar", viking.age() == 40
}
Что если тело пустое? Spring отдаст HTTP 400 (HttpMessageNotReadableException). Если в JSON будет нераспознанное поле — по умолчанию проигнорируется.

2. Аннотации Spring Core (бины и компоненты)

@SpringBootApplication

На главном классе VikingDemoApplication.

Три аннотации в одной:

  • @Configuration — класс с настройками Spring
  • @EnableAutoConfiguration — включить «магию» Spring Boot (он сам настроит Web, JDBC, H2 и т.д. по тому, что в зависимостях)
  • @ComponentScan — найти все классы с @Component, @Service, @Repository, @RestController в текущем пакете и подпакетах
Запомни: без неё Spring Boot ничего не найдёт и приложение не поднимется.

@Service / @Repository / @Component / @RestController

@Service — на VikingService, VikingLambdaService, VikingFactory через @Component.
@Repository — на VikingRepository, EquipmentItemRepository, VikingStorage.
@Component — на VikingMapper, VikingListener.

Все четыре говорят Spring: «создай мне экземпляр этого класса и держи его в IoC-контейнере как singleton». Различие — смысловое, не функциональное: 

@Component     // самое общее — «это бин»
@Service       // бизнес-логика
@Repository    // работа с БД (+ автоконвертация SQL-исключений)
@RestController // веб-контроллер с JSON-ответами

Spring при старте находит все классы с любой из этих аннотаций и создаёт по одному экземпляру каждого. Дальше может «инжектить» их в другие бины через конструктор.

Почему именно @Service у сервиса, а не @Component? Технически работает одинаково, но @Service сообщает читателю «это бизнес-логика» — самодокументация кода.

@Autowired

У тебя в VikingService (на конструкторе), в VikingListener.

Просит Spring «подставить сюда нужный бин». Применяется на:

  • Конструкторе (рекомендовано) — Spring передаст бины при создании объекта.
  • Поле — Spring впрыснет напрямую (плохая практика).
  • Сеттере — Spring вызовет сеттер после создания.
@Service
public class VikingService {
    private final VikingFactory vikingFactory;
    private final VikingStorage vikingStorage;

    @Autowired   // ← можно вообще не писать, если конструктор один
    public VikingService(VikingFactory f, VikingStorage s) {
        this.vikingFactory = f;
        this.vikingStorage = s;
    }
}
В новых версиях Spring @Autowired на конструкторе можно не писать — если у класса единственный конструктор, Spring сам впрыснёт всё что нужно.

3. Транзакции

@Transactional

На методах VikingStorage.save, update, deleteById.

Оборачивает метод в транзакцию БД. Принцип «всё или ничего»:

  • Если метод выполнился до конца без исключений — все SQL-команды коммитятся.
  • Если в середине вылетело RuntimeException — все SQL-команды откатываются.
@Transactional
public Viking save(Viking viking) {
    Integer id = vikingRepository.save(...);        // INSERT в vikings
    for (item : equipment) {
        equipmentItemRepository.save(...);          // INSERT в equipment_items
    }
    // Если упало посередине — оба INSERT откатятся, БД останется чистой.
}
Тонкость: по умолчанию транзакция откатывается только при RuntimeException. На checked-исключения — нет, если не указать rollbackFor.
Зачем тут: без транзакции при ошибке записи equipment у нас бы остался викинг без снаряжения — несогласованное состояние БД.

4. Swagger / OpenAPI (только документация)

Эти аннотации не влияют на работу — они только заполняют веб-страницу Swagger UI на /swagger-ui.html. Если их убрать, эндпоинты будут работать так же, но без описаний.

@Tag(name = "...", description = "...")

На классе контроллера.

Группирует эндпоинты в Swagger UI под общим заголовком. У нас все методы попадают в группу «Vikings».

@Operation(summary = "...", operationId = "...")

На каждом методе контроллера.

Краткое описание эндпоинта. summary — текст для заголовка, operationId — уникальный ID операции в OpenAPI-спецификации.

@ApiResponses({ @ApiResponse(responseCode = "...", description = "...") })

На методах контроллера, описывает возможные коды ответа.
@ApiResponses({
    @ApiResponse(responseCode = "200", description = "Викинг обновлён"),
    @ApiResponse(responseCode = "404", description = "Викинг не найден")
})

Это не валидация! Просто описание для документации. Реальный код ответа задаёт сам метод через ResponseEntity или исключения.

@Schema(description = "...", example = "...")

На полях Viking, на самом record-е.

Описывает поле модели в Swagger: что это, какой пример значения, какого типа.

public record Viking(
    @Schema(description = "Имя викинга", example = "Bjorn")
    String name,
    ...
)

@ArraySchema(...)

На поле equipment в Viking.

То же что @Schema, но для коллекций — описывает массив целиком и тип элементов.

5. Стандартный Java (не Spring)

@Override

В VikingMapper, в record-ах при переопределении методов (если бы были).

Проверка для компилятора: «этот метод должен точно перекрывать метод родителя/интерфейса». Если опечатка в имени или сигнатуре — компилятор выдаст ошибку. 

@Override
public String getModifierName() { return "Снежные ягоды"; }

// если опечатаешься: getModifierNamE() — будет ошибка компиляции,
// потому что нет такого метода в родителе
На рантайм не влияет. Это страховка от опечаток. Желательно ставить везде, где метод что-то перекрывает.

Памятка «куда смотреть в защите»

Если препод тыкает в любую @... — спрашивай себя:

  1. Из какого она пакета? org.springframework.* — Spring, io.swagger.* — только документация, java.lang.* — стандарт.
  2. На чём стоит? На классе → создаёт бин или метаданные класса. На методе → влияет на HTTP-маппинг / транзакцию. На параметре → говорит откуда брать значение.
  3. Что произойдёт, если убрать? Это лучший способ понять смысл аннотации.
