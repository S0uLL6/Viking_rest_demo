Практикум Spring Web — построчный разбор изменений

Задание: добавить в REST API три эндпоинта — добавление конкретного викинга (POST), удаление викинга по id (DELETE) и полная перезапись параметров (PUT). В исходном проекте препода был только GET /api/vikings и POST /api/vikings/post (создавал случайного через GUI-листенер). Чтобы появилось три новых эндпоинта, пришлось не только добавить методы в контроллер, но и доработать модель и слои ниже.

Что изменилось — по файлам

  • ИЗМ model/Viking.java — добавлено поле id
  • ИЗМ repository/VikingMapper.java — маппинг id в обе стороны
  • ИЗМ repository/VikingRepository.java — новый метод update(int, VikingEntity)
  • ИЗМ repository/VikingStorage.java — save теперь возвращает Viking с id, добавлен update
  • ИЗМ service/VikingService.java — добавлены create и update
  • ИЗМ controller/VikingController.java — три новых эндпоинта
  • ИЗМ service/VikingFactory.java — поправлен под новый конструктор Viking (с id)
Принцип архитектуры: в проекте есть слои Controller → Service → Storage → Repository (JDBC). Каждый отвечает за своё. Когда добавляем эндпоинт, цепочка тянется сверху вниз: контроллер парсит HTTP, сервис — бизнес-логика, storage — транзакция и согласованность таблиц, repository — конкретные SQL. Не сваливать всё в контроллер — это и есть ООП «по уму».

1. model/Viking.java — добавлено поле id

Зачем нужно было менять: в оригинале Viking не имел id. GET возвращал список без id — значит клиент не мог сослаться на конкретного викинга при DELETE/PUT (по чему его удалять?). Это был баг исходного API. Добавил id первым полем record-а.
№КодЧто делает
8–22 @Schema(description = "Модель викинга") public record Viking( @Schema(description = "Идентификатор...") Integer id, @Schema(description = "Имя викинга", example = "Bjorn") String name, ... ) {} Record (Java 14+) — это «неизменяемый» класс. Объявил Integer id первым полем (а не int!) — потому что при создании викинга id ещё не известен, его сгенерирует БД, и в это время поле должно быть null. Примитивный int не может быть null.

Аннотации @Schema — для Swagger: в документации API будут описания и примеры.

2. repository/VikingMapper.java — маппинг id в обе стороны

Что было: в toVikingEntity id всегда писался как null; в toViking id вообще пропадал (он не передавался в DTO).
Что стало: id честно прокидывается туда и обратно.
№КодЧто делает
15–24 public VikingEntity toVikingEntity(Viking viking) { return new VikingEntity( viking.id(), // ← было: null viking.name(), viking.age(), viking.heightCm(), viking.hairColor(), viking.beardStyle(), "" ); } Берём DTO Viking (из REST) и превращаем в VikingEntity (для записи в БД через JDBC). Теперь если у DTO есть id (например при PUT для UPDATE) — он попадёт в Entity. Если id = null (при POST для INSERT) — БД сгенерирует новый.
42–53 public Viking toViking(VikingEntity entity, ...) { ... return new Viking( entity.id(), // ← добавили entity.name(), entity.age(), entity.heightCm(), entity.hairColor(), entity.beardStyle(), equipment ); } Обратное превращение: из Entity (из БД) собираем DTO для REST. Теперь клиент увидит id в JSON-ответе GET-а.

3. repository/VikingRepository.java — метод update

Это слой, который пишет SQL через JdbcTemplate. Был save (INSERT) и deleteById (DELETE). Не было UPDATE — добавил.

№КодЧто делает
95–117 public int update(int id, VikingEntity viking) { String sql = """ update vikings set name = ?, age = ?, height_cm = ?, hair_color = ?, beard_style = ?, description = ? where id = ? """; return jdbcTemplate.update( sql, viking.name(), viking.age(), viking.heightCm(), viking.hairColor().name(), viking.beardStyle().name(), viking.description(), id ); } Параметризованный SQL — ? на месте значений, чтобы не было SQL-инъекции. jdbcTemplate.update подставляет аргументы по порядку.

Метод возвращает int — количество затронутых строк. Это важно: если викинга с таким id нет, UPDATE просто ничего не сделает и вернёт 0. По этому 0 наверху (в storage) мы понимаем, что нужно бросить «не найден», а не молча врать клиенту «обновил».
Почему hair_color = ? с .name()? В Java это enum, в БД — VARCHAR. .name() у enum возвращает строковое имя константы ("Red", "BRAIDED" и т.д.).

4. repository/VikingStorage.java — save с возвратом id и update

Storage — это «обёртка» над двумя репозиториями (vikings и equipment_items). Тут согласовываются обе таблицы в одной транзакции.

save — теперь возвращает Viking с проставленным id

№КодЧто делает
31–58 @Transactional public Viking save(Viking viking) { Integer vikingId = vikingRepository.save( vikingMapper.toVikingEntity(viking) ); List<EquipmentItem> equipment = viking.equipment() != null ? viking.equipment() : List.of(); for (EquipmentItem item : equipment) { equipmentItemRepository.save( vikingMapper.toEquipmentItemEntity(vikingId, item) ); } return new Viking( vikingId, viking.name(), viking.age(), viking.heightCm(), viking.hairColor(), viking.beardStyle(), equipment ); }
  1. Сохраняем строку в vikings — БД генерирует id, возвращаем его в переменную vikingId.
  2. Защита от null: если клиент не прислал equipment, обходимся пустым списком вместо NPE.
  3. Цикл сохраняет каждый предмет equipment с привязкой к vikingId.
  4. В конце возвращаем НОВЫЙ Viking с проставленным id. Это то, что увидит клиент в ответе на POST.
@Transactional — если что-то упадёт между шагами 1 и 3, БД откатится: не останется викинга без снаряжения или наоборот.

update — полная перезапись (по требованию практикума)

№КодЧто делает
67–101 @Transactional public Viking update(int id, Viking viking) { int rowsAffected = vikingRepository.update( id, vikingMapper.toVikingEntity(viking) ); if (rowsAffected == 0) { throw new NoSuchElementException( "Викинг с id=" + id + " не найден"); } equipmentItemRepository.deleteByVikingId(id); List<EquipmentItem> equipment = viking.equipment() != null ? viking.equipment() : List.of(); for (EquipmentItem item : equipment) { equipmentItemRepository.save( vikingMapper.toEquipmentItemEntity(id, item) ); } return new Viking(id, viking.name(), ...); } Алгоритм «полной перезаписи»:
  1. UPDATE по таблице vikings. Если 0 строк затронуто — викинга нет, бросаем NoSuchElementException. Это поймает контроллер и превратит в HTTP 404.
  2. Удаляем все старые equipment_items этого викинга.
  3. Вставляем новые equipment_items из тела PUT-запроса.
  4. Возвращаем обновлённого Viking с тем же id.
Почему так: PUT по REST-семантике = «полная замена ресурса». Не PATCH (частичная правка). Поэтому equipment удаляем и вставляем заново — как сказал препод, «перезапись».

Всё в одной транзакции — если на шаге 3 что-то упадёт, equipment не останется потерянным.

deleteById — упростился благодаря FK

№КодЧто делает
120–124 @Transactional public void deleteById(int id) { // equipment_items удалятся каскадом (FK on delete cascade) vikingRepository.deleteById(id); } В schema.sql у equipment_items.viking_id стоит foreign key ... on delete cascade. Это значит: когда удаляем строку из vikings, БД сама удалит все связанные строки из equipment_items. Нам в коде ничего дополнительно делать не нужно.

Сервисный слой просто пробрасывает вызовы в storage. Бизнес-логики (валидаций, нотификаций и т.п.) у нас здесь нет.

POST /api/vikings — добавить конкретного викинга

№КодЧто делает

74–78 @Operation(summary = "Добавить конкретного викинга", ...) @ApiResponses({ @ApiResponse(responseCode = "201", ...) }) Аннотации Swagger — описание метода в документации. Не влияют на работу, только на отображение.

  2. Формируем URL нового ресурса.
  3. Возвращаем HTTP 201 Created + заголовок Location: /api/vikings/4 + тело с самим Viking. Это «правильный» REST-ответ на POST.

DELETE /api/vikings/{id} — удалить викинга

№КодЧто делает
86 @DeleteMapping("/{id}") Маппинг на DELETE /api/vikings/{что-то}. {id} — путевой параметр.
92 public ResponseEntity<Void> deleteViking(@PathVariable int id) @PathVariable — Spring подставит сюда значение из URL. Если запрос DELETE /api/vikings/5, то id = 5.
ResponseEntity<Void> — потому что мы вернём 204 без тела.
93–95 vikingService.deleteById(id); return ResponseEntity.noContent().build(); Удаляем (если викинга нет — БД молча ничего не сделает). Возвращаем HTTP 204 No Content — стандарт для успешного DELETE без тела ответа.
Почему DELETE не возвращает 404, если викинга нет?
REST допускает оба варианта. Я выбрал «всегда 204», потому что идемпотентность: повторный DELETE того же id должен давать тот же результат. Если первый раз вернёт 204, а второй раз 404 — это нарушает идемпотентность. Препод спросит — отвечай так.

PUT /api/vikings/{id} — полностью перезаписать викинга

№КодЧто делает
98 @PutMapping("/{id}") Маппинг на PUT /api/vikings/{id}.
105 public Viking updateViking( @PathVariable int id, @RequestBody Viking viking) Два источника данных: id — из URL, тело — из JSON. Spring разберёт оба автоматически.
Возвращаем напрямую Viking (не ResponseEntity) — потому что код ответа по умолчанию 200 OK, нас устраивает.
107–111 try { return vikingService.update(id, viking); } catch (NoSuchElementException e) { throw new ResponseStatusException( HttpStatus.NOT_FOUND, e.getMessage()); } Ловим NoSuchElementException из service (он был брошен в storage, если викинг не найден). Преобразуем в ResponseStatusException с кодом HTTP 404 Not Found — Spring сам отправит правильный ответ.

Почему не создаём новый при PUT? По REST-семантике PUT может «создать или обновить», но в нашей системе id генерирует БД, и клиент не должен сам присваивать id. Поэтому если id нет — это ошибка клиента (404), а не команда «создай заново».

8. Обновление GUI после REST-операций

Проблема: REST-эндпоинты пишут данные в БД, но JTable в Swing-окне об этом не знает — таблица показывает старые данные до перезапуска. Препод спросил «а где обновление GUI?» — пришлось дорабатывать.

Решение: после каждой REST-операции контроллер вызывает соответствующий метод листенера, который точечно обновляет одну строку таблицы. Никакой перерисовки всего списка.

Цепочка вызовов

ШагГдеЧто делает
1 VikingController После операции с БД вызывает vikingListener.onCreated(viking) / onDeleted(id) / onUpdated(viking)
2 VikingListener Оборачивает действие в SwingUtilities.invokeLater(...) — перевод в поток EDT
3 VikingDesktopFrame Вызывает соответствующий метод tableModel: addViking / removeByVikingId / replaceViking
4 VikingTableModel Меняет внутренний список data и оповещает таблицу через fireTableRowsInserted/Deleted/Updated
5 JTable Получает уведомление и перерисовывает ТОЛЬКО изменившуюся строку

Фрагменты кода по шагам (на примере POST)

Шаг 1 · VikingController.java
@PostMapping
public ResponseEntity<Viking> createViking(@RequestBody Viking viking) {
    Viking created = vikingService.create(viking);   // сохранили в БД, получили объект с id
    vikingListener.onCreated(created);                // → шаг 2
    URI location = URI.create("/api/vikings/" + created.id());
    return ResponseEntity.created(location).body(created);
}
Шаг 2 · VikingListener.java
public void onCreated(Viking viking) {
    if (gui != null) {
        SwingUtilities.invokeLater(() -> gui.addNewViking(viking));
    }
}
Тут происходит переход из потока Tomcat в EDT — invokeLater ставит лямбду в очередь Swing.
Шаг 3 · VikingDesktopFrame.java
public void addNewViking(Viking viking) {
    tableModel.addViking(viking);   // → шаг 4
}
Frame ничего не решает сам — просто проброс в модель.
Шаг 4 · VikingTableModel.java
public void addViking(Viking viking) {
    int row = data.size();
    data.add(viking);                       // добавили в список
    fireTableRowsInserted(row, row);        // → JTable: «вставлена строка row»
}
Шаг 5 · JTable (встроенный механизм Swing)
Кода тут нет — это код самого Swing. JTable слушает события модели. Метод fireTableRowsInserted вызывает tableChanged у JTable, она запрашивает значения только для новой строки через getValueAt(row, col) и рисует. Остальные строки не трогаются.

Аналогично для DELETE и PUT — меняются только методы

ШагPOSTDELETEPUT
1 (Controller) vikingListener.onCreated(created) vikingListener.onDeleted(id) vikingListener.onUpdated(updated)
2 (Listener) gui.addNewViking(viking) gui.removeViking(id) gui.replaceViking(viking)
3 (Frame) tableModel.addViking(viking) tableModel.removeByVikingId(id) tableModel.replaceViking(viking)
4 (TableModel) fireTableRowsInserted(row, row) fireTableRowsDeleted(i, i) fireTableRowsUpdated(i, i)

Что делают конкретные методы

ОперацияМетод листенераЧто происходит с таблицей
POST onCreated(Viking v) tableModel.addViking(v) — добавляет строку в конец, fireTableRowsInserted
DELETE onDeleted(int id) tableModel.removeByVikingId(id) — ищет по id, удаляет, fireTableRowsDeleted
PUT onUpdated(Viking v) tableModel.replaceViking(v) — ищет по id, заменяет объект, fireTableRowsUpdated

Код в VikingTableModel.java — поиск по id

№КодЧто делает
31–40 public void removeByVikingId(int id) { for (int i = 0; i < data.size(); i++) { Viking v = data.get(i); if (v.id() != null && v.id() == id) { data.remove(i); fireTableRowsDeleted(i, i); return; } } } Пробегаем по списку, находим викинга с нужным id, удаляем из списка и оповещаем JTable что удалилась строка с индексом i. JTable перерисовывает только её.
43–55 public void replaceViking(Viking viking) { if (viking.id() == null) return; for (int i = 0; i < data.size(); i++) { Viking v = data.get(i); if (v.id() != null && v.id().equals(viking.id())) { data.set(i, viking); fireTableRowsUpdated(i, i); return; } } } То же самое, но с заменой: data.set(i, viking) кладёт новый объект на то же место. fireTableRowsUpdated — JTable перерисует только эту строку, не двигая выделение и прокрутку.
Зачем fireTableRowsXxx? VikingTableModel наследуется от AbstractTableModel. JTable сама не знает что изменилось в данных — она получает оповещение от модели. fireTableRowsInserted/Deleted/Updated и есть это оповещение: «строки с такого по такой индекс добавились/удалились/обновились». Без вызова fire... JTable будет показывать старое.
Зачем SwingUtilities.invokeLater? REST-запросы Spring обрабатывает в потоке Tomcat. Swing требует чтобы любые изменения GUI выполнялись в специальном потоке EDT (Event Dispatch Thread). Прямой вызов из Tomcat-потока может вызвать гонки данных и нестабильные баги. invokeLater ставит задачу в очередь EDT — Swing сам её выполнит в безопасном потоке.
Сравнение точечного и общего подхода:
Общий refresh (был раньше) Точечное обновление (сейчас)
Что делает очищает таблицу + читает всю БД + добавляет всех меняет одну строку
Походов в БД 1 (findAll) 0
JTable перерисовывает всё только одну строку
Выделение/прокрутка сбрасывается сохраняется
Когда лучше если кто-то ещё пишет в БД не через наш API штатный сценарий, всё идёт через наш контроллер

Вопросы препода по этой теме

  • Как у тебя обновляется таблица?
    «После REST-операции контроллер вызывает соответствующий метод листенера: onCreated, onDeleted или onUpdated. Листенер через SwingUtilities.invokeLater переводит выполнение в EDT и дёргает метод frame-а. Frame через tableModel точечно меняет одну строку и оповещает JTable через fireTableRowsXxx.»
  • Почему через VikingListener, а не из контроллера напрямую?
    «Контроллер не должен знать про GUI — он работает только с REST. Листенер специально создан как мост между REST-слоем и Swing-слоем, у него уже была ссылка на главное окно.»
  • Зачем SwingUtilities.invokeLater?
    «REST приходит в потоке Tomcat, а Swing требует EDT. invokeLater безопасно ставит задачу в очередь EDT.»
  • Почему точечно, а не общий refresh?
    «Эффективнее: одна правка вместо очистки и перечитки всей БД. Сохраняется состояние UI (выделение, прокрутка). Для большой таблицы это критично, для маленькой — просто чище.»
  • Что если викинга с таким id нет в таблице (при PUT/DELETE)?
    «Метод ищет в data по id и если не находит — просто возвращается. Это safe-default: значит таблица и БД рассинхронизированы. В нашем сценарии этого не должно случаться, потому что все правки идут через наш контроллер.»

Возможные вопросы препода и ответы

Про архитектуру

  • Зачем нужен Storage между Service и Repository? Чтобы согласовывать две таблицы (vikings и equipment_items) в одной транзакции. VikingRepository и EquipmentItemRepository сами по себе ничего не знают друг о друге.
  • Что такое @Transactional? Аннотация Spring, которая оборачивает метод в транзакцию: либо все SQL внутри выполнятся успешно, либо все откатятся. Защищает от частичных записей при ошибках.
  • Где валидация входящих данных? Сознательно не делал — препод не требовал. Можно добавить @Valid и аннотации @NotBlank, @Min на полях Viking — это Bean Validation (JSR-380).

Про HTTP-коды

  • Почему POST возвращает 201, а не 200? 201 Created — это стандартный код для успешного создания нового ресурса. 200 OK — для просто «всё ок» без указания на создание. Препод любит когда «по уму».
  • Почему DELETE 204? 204 No Content — успешно, но в теле ответа нечего возвращать. Это правильный код для DELETE без полезной нагрузки.
  • Почему PUT 200, а не 204? Потому что мы возвращаем тело с обновлённым викингом — клиенту полезно видеть «вот что в итоге сохранилось». При пустом теле было бы 204.
  • Почему PUT может вернуть 404? Если викинга с таким id нет — обновлять нечего. Это ошибка клиента (он сослался на несуществующий ресурс).

Про Spring-аннотации

  • @RestController vs @Controller? RestController = Controller + @ResponseBody на всех методах. Возвращаемые объекты автоматически сериализуются в JSON.
  • Как работает @RequestBody? Spring читает тело HTTP-запроса (JSON), берёт Jackson и десериализует в указанный Java-тип. Поля сопоставляются по именам.
  • Как Spring сопоставляет {id} в пути с @PathVariable int id? По имени — оба называются id. Если бы переменная называлась иначе, нужно было бы писать @PathVariable("id") int vikingId.

Про PUT vs PATCH

  • В чём разница? PUT — полная замена ресурса (тело должно содержать весь объект целиком). PATCH — частичное обновление (тело содержит только изменения).
  • Почему здесь PUT? Препод просил «перезапись параметров» — это слово соответствует семантике PUT. Если бы было сказано «изменить отдельные поля» — был бы PATCH.
  • Что с equipment при PUT? Полная замена: старые equipment_items удаляются, новые из тела вставляются. Если в теле equipment = пустой массив — у викинга вообще не будет снаряжения после обновления. Это и есть «полная перезапись».
